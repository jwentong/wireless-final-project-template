# 无线通信技术期末项目实验报告

## 摘要

本项目完成了一个端到端无线通信基带仿真系统。系统以 `Test.txt` 中的 UTF-8 文本为输入，经过信源编码、PN 序列 XOR 扰码、信道编码、帧构建、调制、无线信道、帧同步、解调、信道译码、解扰、信源译码等模块，最终恢复为 `results/received.txt`，并生成误码率、帧错误率、文本恢复率、校验结果、星座图、BER-SNR 曲线和同步相关峰值图。

基础链路采用 Gray 编码 QPSK 调制、AWGN 信道、重复 3 次信道编码和前导序列相关同步。在 `SNR = 12 dB`、`seed = 2026`、`QPSK`、`AWGN` 条件下，系统实现了 `Test.txt` 到 `received.txt` 的完全恢复，`BER = 0.0`，`FER = 0.0`，`text_match_rate = 1.0`，`checksum_pass = true`。在扩展部分，本项目进一步实现了 Rayleigh 平坦衰落与均衡、OFDM、二分支 MRC 接收分集、卷积码与 Viterbi 译码、自适应调制以及本地 Web 可视化界面，覆盖老师要求的 Level 1、Level 2、Level 3 内容。

关键词：无线通信；基带仿真；QPSK；AWGN；帧同步；信道编码；OFDM；Rayleigh 衰落；AI 辅助编程

## 1. 项目背景与实验目的

### 1.1 项目背景

本课程期末项目要求使用 AI 辅助编程完成一个无线通信技术综合实践项目。相比传统闭卷考试，本项目更强调对完整通信链路的理解、工程实现能力、测试验证能力和结果分析能力。项目不只是写出单个调制或解调函数，而是要把文本文件真正送入通信系统，经历发送端、信道和接收端的完整处理流程，并用实验指标证明接收结果是否可靠。

本项目围绕“文本文件的无线基带传输仿真”展开。发送端从 `Test.txt` 读取文本，将文本转换为比特流并进行扰码、编码、组帧和调制；信道端加入 AWGN 或 Rayleigh 衰落等信道影响；接收端通过同步、解调、译码和解扰恢复原始文本，并输出 `received.txt`。整个系统同时生成可解释的中间指标与图像，便于课程验收和答辩说明。

### 1.2 实验目的

本实验的主要目标如下：

1. 理解端到端无线通信链路的组成，包括信源编码、扰码、信道编码、组帧、调制、信道、同步、解调、译码和信源恢复。
2. 实现 `Test.txt` 到 `received.txt` 的可靠文本传输，并验证接收文件与发送文件的一致性。
3. 掌握 QPSK Gray 映射、AWGN 信道建模、前导序列同步、重复码纠错、CRC 校验等核心通信模块。
4. 通过 BER、FER、文本恢复率、checksum 结果和同步误差等指标评价系统性能。
5. 通过星座图、BER-SNR 曲线和同步峰值图分析 SNR 对系统性能的影响。
6. 完成 Level 3 扩展模块，包括 Rayleigh 衰落、均衡、OFDM、分集、卷积码、Viterbi 译码、自适应调制和 Web 可视化界面。
7. 规范记录 AI 辅助编程过程，说明 AI 生成内容、人为修改内容和测试验证过程。

## 2. 系统总体设计

### 2.1 系统整体流程

系统的端到端处理链路如下：

```text
Test.txt
-> Source Encode
-> Scramble / Encrypt
-> Channel Encode
-> Frame Build
-> QPSK Modulate
-> Wireless Channel
-> Synchronization
-> QPSK Demodulate
-> Channel Decode
-> Descramble / Decrypt
```

该流程严格模拟了无线基带系统的基本结构。发送端负责把离散文本信息转换成适合信道传输的调制符号；信道端加入噪声、随机前缀或衰落；接收端在不知道帧起点的情况下利用前导序列完成同步，然后恢复比特和文本。

## 3. 关键通信原理

### 3.1 信源编码与信源译码

信源编码模块将输入文本按 UTF-8 编码转换为字节序列，再把每个字节转换为 8 位 big-endian 比特。例如一个字节的最高有效位先进入 bitstream。接收端根据帧中记录的原始 payload 长度去除多余填充，再按 8 bit 重新组合字节，并用 UTF-8 解码恢复文本。

采用 UTF-8 的原因是它可以同时表示英文、中文和符号，符合本项目测试文本包含中文内容的要求。长度字段记录的是信源编码后的原始 bit 数，不依赖具体文本内容，因此可以处理不同长度的输入文件。

### 3.2 扰码与解扰

本项目采用 PN 序列 XOR 扰码。系统根据 `seed` 生成与 payload 等长的伪随机 0/1 序列，然后对信源比特逐位异或：

```text
scrambled_bit = source_bit XOR pn_bit
```

XOR 运算具有自反性，因此接收端使用相同 seed 生成同一 PN 序列，再执行一次 XOR 即可恢复原始比特：

```text
source_bit = scrambled_bit XOR pn_bit
```

该模块主要用于模拟通信系统中的扰码或简单加密过程，使传输比特更加随机化。它不是强密码学意义上的安全加密，但能够体现“可逆扰码”和“随机种子控制”的通信工程思想。

### 3.3 信道编码与译码

基础链路采用重复 3 次编码。编码器把每个信息比特重复 3 次：

```text
0 -> 000
1 -> 111
```

译码器对每 3 位进行多数判决。如果 3 位中至少 2 位为 1，则判为 1；否则判为 0。该码的码率为 `1/3`，即传输效率降低为原来的三分之一，但能够纠正每组三位中的单个错误。

Level 3 扩展中还实现了卷积码与 Viterbi 译码。卷积码使用约束长度为 3、生成多项式为 7 和 5 八进制的 rate-1/2 编码结构，接收端采用硬判决 Viterbi 算法搜索最可能的状态路径。相比重复码，卷积码具有更好的编码效率和更系统的纠错机制。

### 3.4 帧结构设计

系统帧结构为：

```text
preamble(64) | payload_length(32) | tx_payload_length(32) | payload | crc32(32)
```

各字段含义如下：

| 字段 | 作用 |
|---|---|
| `preamble` | 固定前导序列，用于接收端同步 |
| `payload_length` | 原始信源 payload bit 数，用于最终恢复和去除 padding |
| `tx_payload_length` | 信道编码后的 transmitted payload bit 数，用于准确解析帧 |
| `payload` | 经过扰码和信道编码后的数据 |
| `crc32` | 对原始 payload bitstream 计算 CRC32，用于接收端校验 |

这里同时保存 `payload_length` 和 `tx_payload_length` 是一次 mock 测试后确定的设计修订。原始 payload 长度用于满足 PRD 中“按原始文本比特长度恢复”的语义，而 transmitted payload 长度用于解决信道编码后 payload 长度变长的问题。这样既能保证帧解析正确，又能保证接收端恢复原始文本时不会受到编码冗余或 QPSK padding 的影响。

### 3.5 QPSK 调制与解调

基础调制方式为 Gray 编码 QPSK，每个符号携带 2 bit。映射关系为：

| 比特 | 星座点 |
|---|---|
| `00` | `(1+j)/sqrt(2)` |
| `01` | `(-1+j)/sqrt(2)` |
| `11` | `(-1-j)/sqrt(2)` |
| `10` | `(1-j)/sqrt(2)` |

`1/sqrt(2)` 归一化使平均符号功率约为 1，便于后续 AWGN 噪声功率计算。解调时采用硬判决：根据接收点所在象限判定最接近的星座点，再映射回对应比特。若进入 QPSK 的比特数为奇数，调制器会补一个 0；接收端根据帧长度字段删除 padding。

### 3.6 AWGN 信道模型

AWGN 信道可以表示为：

```text
y = x + n
```

其中 `x` 为发送符号，`n` 为复高斯白噪声。SNR 定义为平均符号功率与噪声功率之比：

```text
noise_power = signal_power / 10^(SNR_dB / 10)
```

由于噪声是复数，实部和虚部分别加入方差为 `noise_power / 2` 的高斯噪声。系统使用固定 `seed` 控制随机数生成，使实验结果可复现。

### 3.7 帧同步原理

发送端在真实帧前插入 0 到 128 个随机 QPSK 符号作为前缀，接收端不能假设帧从第一个符号开始。同步模块使用已知前导序列进行滑动相关检测：

```text
correlation(k) = |sum(received[k : k+L] * conj(preamble))|
```

当候选窗口与前导序列对齐时，相关值达到峰值。因此接收端选择相关峰值最大的位置作为帧起点，并记录 `sync_start_index`、`sync_expected_offset` 和 `sync_error_symbols`。本项目基础实验中同步误差为 0，说明接收端成功找到了帧起点。

### 3.8 Level 3 高级通信原理

Rayleigh 平坦衰落模型为：

```text
y = h x + n
```

其中 `h` 为复高斯衰落系数。接收端在 perfect CSI 假设下使用一抽头均衡：

```text
x_hat = y / h
```

OFDM 模块将调制符号填入 64 个子载波，通过 IFFT 生成时域波形，并添加长度为 16 的循环前缀。循环前缀可以缓解多径信道中的符号间干扰，本项目中主要用于展示 OFDM 的基本收发流程。

MRC2 分集使用两个独立 Rayleigh 接收分支，并按信道共轭加权合并，提高接收可靠性。自适应调制根据 SNR 选择 BPSK、QPSK 或 16-QAM：低 SNR 使用更稳健的 BPSK，中等 SNR 使用 QPSK，高 SNR 使用频谱效率更高的 16-QAM。

### 4.3 各模块代码实现

#### 4.3.1 信源编码模块

`src/source.py` 负责读取 UTF-8 文本并转换为 bitstream。实现时没有假设文本内容，也没有硬编码 `Test.txt` 的答案。只要输入能被 UTF-8 表示，就可以按字节转换为比特，并在接收端按原长度恢复。

#### 4.3.2 扰码模块

`src/crypto.py` 使用 NumPy 随机数生成器产生 PN 序列，并执行 XOR。发送端和接收端使用同一个 `seed`，因此扰码和解扰完全对称。Web 界面中还提供 demo 模式关闭扰码，便于教学展示扰码前后的差异。

#### 4.3.3 信道编码模块

基础版本在 `src/channel_coding.py` 中实现重复 3 次编码和多数判决译码。高级版本在 `src/convolutional.py` 中实现卷积编码和 Viterbi 译码。系统通过 `--coding` 参数选择编码方式，因此基础链路与高级链路可以共存。

#### 4.3.4 帧构建与解析模块

`src/framing.py` 将前导序列、长度字段、payload 和 CRC 组合为帧。解析时先定位前导，再读取长度字段，随后按 `tx_payload_length` 提取编码后的 payload，最后用 `payload_length` 恢复原始信源比特。CRC 的作用是判断恢复出的原始 payload 是否可信。

#### 4.3.5 调制与解调模块

`src/modulation.py` 实现 QPSK Gray 映射、BPSK、16-QAM 和自适应调制。基础验收使用 QPSK，扩展实验中 `--mod adaptive` 会根据 SNR 选择实际调制方式，并在 `metrics.json` 中同时记录 `requested_modulation` 和 `effective_modulation`。

#### 4.3.6 信道模块

`src/channel.py` 支持 AWGN 和 Rayleigh 信道。AWGN 负责加入复高斯噪声；Rayleigh 模式会引入复衰落系数，并可配合均衡或分集合并使用。随机前缀也在信道阶段插入，用于验证同步模块不是依赖已知帧起点。

#### 4.3.7 同步模块

`src/synchronization.py` 使用前导序列相关峰值搜索帧起点。模块输出检测到的起点和相关值序列，后续由 `pipeline.py` 生成同步峰值图。基础实验中 `sync_start_index = 109`，`sync_expected_offset = 109`，`sync_error_symbols = 0`。

#### 4.3.8 指标统计与可视化模块

`src/pipeline.py` 是系统调度核心。它串联全部模块，保存接收文本，生成 `metrics.json`，并输出星座图、BER-SNR 曲线和同步峰值图。指标包括：

| 指标 | 含义 |
|---|---|
| `ber` | 接收处理后的 bit error rate |
| `fer` | 帧错误率或帧失败标志 |
| `text_match_rate` | 接收文本与原文本的匹配率 |
| `checksum_pass` | CRC/checksum 是否通过 |
| `sync_error_symbols` | 同步误差，单位为符号 |
| `failure_reason` | 失败原因说明，成功时为 null |

### 4.4 Web 可视化实现

`web_app.py` 提供本地 Web 仿真界面。运行命令为：

```bash
python web_app.py --host 127.0.0.1 --port 8000
```

Web 页面支持输入自定义文本、调节 SNR 和 seed、选择 AWGN/Rayleigh 信道、选择扰码和信道编码模式、选择高级调制与 OFDM/分集选项。页面调用与 CLI 相同的 `run_pipeline()`，因此图形界面不会绕过真实通信链路。每次运行后，页面展示接收文本、指标、星座图、BER-SNR 曲线、同步峰值图和每个通信阶段的 trace 信息。

### 6.3 BER-SNR 性能分析

多 SNR 实验显示，在低 SNR 条件下，接收星座点分布较散，硬判决误码较多，导致 BER 较高、checksum 失败、文本恢复率下降。随着 SNR 提高，噪声功率降低，星座点逐渐收敛到理想 QPSK 星座位置附近，误判概率下降。

在 `awgn-coded` 场景中，0 dB、3 dB 和 6 dB 下 BER 约为 0.52，文本无法恢复；9 dB 及以上 BER 降为 0，文本恢复率达到 1.0，checksum 通过。这说明重复码和同步模块在中高 SNR 条件下能有效保障链路可靠性。

### 6.4 星座图分析

`results/constellation.png` 展示了同步后的接收星座点。在 12 dB AWGN 条件下，星座点集中在 QPSK 的四个理想点附近，虽然存在噪声扩散，但没有明显跨越判决边界，因此硬判决解调能够得到正确比特。

如果降低 SNR，噪声方差增大，星座点云会向中心和相邻象限扩散。当点云越过实轴或虚轴判决边界时，就会发生 bit error；如果错误超过重复码的纠错能力，就会导致 CRC 失败或文本恢复失败。

### 6.5 同步峰值图分析

`results/sync_peak.png` 展示了前导相关值随候选起点变化的曲线。正确帧起点处出现最大峰值。本次基础实验中随机前缀长度为 109 个符号，接收端检测到的 `sync_start_index` 也是 109，因此同步误差为 0。

同步模块是本项目的重要部分，因为真实接收端不能直接知道帧起点。通过相关峰值搜索，可以把帧同步问题转化为一个最大值检测问题。

### 6.6 文本恢复结果分析

基础实验中 `text_match_rate = 1.0`，`checksum_pass = true`，说明恢复文本与原始文本完全一致。若低 SNR 下出现乱码或文本缺失，可能原因包括：

1. 同步峰值检测错误，导致后续帧解析错位。
2. QPSK 硬判决产生过多误码。
3. 重复码多数判决无法纠正每组三位中的多个错误。
4. CRC 检查失败，说明原始 payload bitstream 已损坏。
5. UTF-8 字节序列被破坏，导致解码失败或部分字符错误。

本系统对这些失败情况不会直接崩溃，而是在 `metrics.json` 中记录 BER、FER、文本恢复率、checksum 结果和 `failure_reason`。

## 8. 问题分析与改进方向

### 8.1 开发过程中遇到的问题

1. 原始 payload 长度与信道编码后 payload 长度混淆，导致帧解析存在风险。
2. QPSK 输入 bit 数可能为奇数，需要 padding 并在接收端正确删除。
3. 接收端不能假设帧起点已知，需要实现前导同步。
4. CRC/checksum 的覆盖范围需要统一，否则会出现发送端和接收端校验不一致。
5. 低 SNR 下重复码纠错能力有限，文本可能无法恢复。
6. 中文 UTF-8 文本对 bit 长度和字节边界要求严格，一旦误码就可能出现解码失败。

### 8.2 解决方法

针对上述问题，本项目采取了以下措施：

1. 在帧结构中同时记录 `payload_length` 和 `tx_payload_length`。
2. 在调制阶段处理 QPSK padding，并通过帧长度字段恢复真实 bit 数。
3. 使用前导序列相关峰值搜索实现同步，并记录同步误差。
4. 明确 CRC 范围为原始 payload bitstream，并在指标中记录 `frame_crc_scope`。
5. 使用重复 3 次编码提高基础链路鲁棒性，并扩展卷积码/Viterbi。
6. 对失败情况输出 `failure_reason` 和指标，而不是让程序直接崩溃。

### 8.3 系统不足

当前系统仍存在一些不足：

1. 基础链路使用硬判决解调，没有实现软判决译码。
2. 重复码编码效率较低，码率只有 `1/3`。
3. Rayleigh 均衡使用 perfect CSI，真实系统中需要信道估计。
4. OFDM 模块主要用于流程展示，没有深入实现复杂多径信道下的同步和信道估计。
5. Web 界面是本地演示界面，不是完整的生产级前后端系统。

### 8.4 后续改进方向

后续可以从以下方向继续改进：

1. 引入软判决 Viterbi 或 LDPC/Turbo 编码，提高低 SNR 性能。
2. 实现导频辅助信道估计，使 Rayleigh 均衡更接近真实系统。
3. 扩展 Rician、多径频率选择性衰落信道。
4. 完善 OFDM 的频偏估计、定时同步和子载波均衡。
5. 加入更系统的自适应编码调制策略，根据 BER 或信道估计结果动态调整参数。
6. 将 Web 界面扩展为更完整的教学平台，显示每一级波形、频谱和误码变化。

## 9. 总结

本项目完成了一个从 `Test.txt` 到 `received.txt` 的端到端无线通信基带仿真系统。基础链路包括 UTF-8 信源编解码、PN 扰码、重复码信道编码、帧构建、Gray 编码 QPSK、AWGN 信道、前导序列同步、硬判决解调、信道译码、解扰、文本恢复和指标统计。在 `SNR = 12 dB` 的基础验收条件下，系统实现了文本完全恢复，BER、FER 均为 0，checksum 通过。

在扩展部分，本项目进一步实现了 Rayleigh 衰落与均衡、OFDM、MRC2 分集、卷积码与 Viterbi 译码、自适应调制和 Web 可视化界面，覆盖 Level 1、Level 2、Level 3 要求。通过 mock 测试、单元测试、公共测试和多 SNR 实验，项目验证了核心模块的正确性，也展示了低 SNR 下无线通信链路的性能退化规律。

通过本项目，我加深了对无线通信系统完整链路的理解：一个可靠接收结果不仅依赖调制解调，还依赖帧结构、长度字段、同步、编码、校验和工程测试等多个环节的协同。AI 辅助提高了开发效率，但最终仍需要用通信原理和实验结果确认每个模块的正确性。
