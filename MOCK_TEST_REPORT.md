## 3. 发现的设计缺陷 / 风险

**缺陷（高风险）：帧头 `length` 字段未受前向纠错保护。**

- 根因：PRD 规定的固定链路顺序为 `… → Channel Encode → Frame Build → Modulate`，即**信道编码在帧封装之前**。
  因此帧的前导、`length` 字段与 CRC 是在信道编码**之后**才拼接进帧的，这些控制字段**不在 FEC 保护范围内**。
- 后果：低 SNR 下，只要一个比特错误落在 32-bit 的 `orig_len`/`coded_len` 字段上，接收端就会按错误长度切片，
  导致 payload 边界错位、整帧解析失败（mock-6 中 4 dB 有 18/20、6 dB 有 11/20 属此类"header 字段损坏"）。
- 影响面：验收基线（SNR≥12 dB）不受影响，但**隐藏验证集若使用较低 SNR**，系统会出现远超预期的失败率，属设计盲点。

（次要风险：payload 在极低 SNR 下的残余误码，主要由 mock 占位用的弱重复码导致，正式实现改用卷积码后改善。）

## 4. 对 DESIGN.md 的修订

针对上述缺陷，对 `DESIGN.md` 第 4.4 节帧结构做出如下**修订**（不改变 PRD 链路顺序）：

> **修订**：帧头的 `orig_len`、`coded_len` 长度字段采用 **3× 重复编码 + 接收端多数判决** 进行轻量保护。
> 每个 32-bit 字段在帧内占 96 bit；接收端先对这 192 bit 做 3-bit 多数表决恢复长度字段，再据此切分 payload 与 CRC。

理由：长度字段是 framing 的"控制平面"，一旦出错代价远高于单个 payload 比特；用极小开销（每帧多 128 bit）
换取对单比特错误的免疫，显著提升低 SNR / 隐藏测试鲁棒性。修订已回填 `DESIGN.md` 附录 A（v0.2）。

## 5. 修订验证（before / after）

运行 `mock/mock_fix_verify.py`（仅新增 header 保护，其余不变）：

| SNR (dB) | 修订前 成功 | 修订前 header 损坏 | 修订后 成功 | 修订后 header 损坏 |
|---|---|---|---|---|
| 4 | 0/20 | 18 | 0/20 | **6** |
| 6 | 7/20 | 11 | **12/20** | **1** |
| 8 | 19/20 | 0 | 19/20 | 0 |
| 10 | 20/20 | 0 | 20/20 | 0 |
| 12 | 20/20 | 0 | 20/20 | 0 |

结论：header 字段损坏在 6 dB 由 11→1、4 dB 由 18→6 **大幅下降**，6 dB 整体成功率由 7→12。
修订有效。剩余失败转为 payload 残余误码，将由正式实现的**卷积码 K=7 + Viterbi**进一步解决。

3. 同步、QPSK、AWGN、源/扰码在 mock 中接口已验证自洽，正式实现保持同接口，进入 TDD。
4. mock 骨架（`mock/`）保留作为设计验证留痕，不参与最终链路。
